# Migrate `ballot-interpreter` from Neon to napi-rs

**Author:** @eventualbuddha

**Status:** `completed`

## Existing Discussion

- [napi-rs LSP issue (now fixed): LSP autocomplete useless with napi macros](https://github.com/napi-rs/napi-rs/issues/944)
- [Fix PR: refactor(napi-derive): expand order](https://github.com/napi-rs/napi-rs/pull/2265)

## Problem

`libs/ballot-interpreter` uses [Neon](https://neon-bindings.com) (v1.1.0) to
bridge its Rust ballot-interpretation logic to Node.js. This has two practical
consequences that create ongoing maintenance burden:

**1. The JS loader and function signatures are hand-maintained.**

`libs/ballot-interpreter/src/bubble-ballot-ts/addon.ts` manually loads the
compiled `.node` binary, manually declares every exported function's signature,
and manually adapts the Rust API surface (e.g., converting `ImageData` objects
into separate width/height/buffer arguments because Neon can't receive rich
types directly). This is boilerplate that must be kept in sync with the Rust
code.

**2. TypeScript types for Rust structs are maintained by hand.**

Every type that crosses the Rust/JS boundary is duplicated: Rust structs are
defined in the crate, and equivalent TypeScript interfaces are hand-written in
`libs/ballot-interpreter/src/bubble-ballot-ts/types.ts` (~350 lines). This
dual-maintenance requirement is error-prone — type drift between Rust and
TypeScript cannot be caught at compile time, only at runtime. Note: the JSON
round-tripping approach chosen here (see Prototype Findings) preserves this
hand-written `types.ts` rather than replacing it with auto-generated types, so
this problem is not fully solved by the migration. napi-rs does offer
`#[napi(object)]` for auto-generating TypeScript interfaces, but that approach
was not viable for this codebase's complex enum types (see Prototype Findings).

**Why change now?**

The codebase already uses napi-rs in `libs/logging-utils`. Having two different
Rust/Node bridge libraries in the same monorepo increases cognitive overhead and
splits tooling knowledge across the team. Consolidating on one library is
desirable. As part of [#8041][issue-8041], I plan to make the ballot-interpreter
API async, which will require some changes. We might as well bundle the changes
together.

[issue-8041]: https://github.com/votingworks/vxsuite/issues/8041

The previously known blocker for adopting napi-rs more broadly — that the
`#[napi]` proc-macro broke `rust-analyzer` LSP completions inside annotated
functions ([issue #944](https://github.com/napi-rs/napi-rs/issues/944), open
since December 2021) — was resolved in
[PR #2265](https://github.com/napi-rs/napi-rs/pull/2265), merged September 20,
2024, and shipped in napi-rs v2.18.0.

## Proposal

Replace Neon with napi-rs in `libs/ballot-interpreter`, following the same
pattern already established in `libs/logging-utils`.

The concrete changes are:

1. **Rust side:** Replace `#[neon::export]` attributes and
   `neon::types::extract::{Json, Error}` usage with `#[napi]` attributes from
   `napi_derive`. Add `build.rs` with `napi_build::setup()`. Update `Cargo.toml`
   to replace `neon` with `napi`, `napi-derive`, and `napi-build`.

2. **Auto-generated loader:** Delete the hand-written `addon.ts` loader. napi-rs
   generates `index.d.ts` (TypeScript declarations) and `index.js`
   (platform-detecting binary loader) at build time. The hand-written `types.ts`
   is kept as the authoritative TypeScript type source — the generated
   `index.d.ts` references it via a `dts-header.d.ts` file.

3. **Build tooling:** Replace `cargo-cp-artifact` with `@napi-rs/cli` (already
   used by `logging-utils`). The output changes from a single `build/addon.node`
   to per-platform binaries like `ballot-interpreter.linux-x64-gnu.node`. This
   doesn't matter in practice since we don't publish the package to NPM.

4. **Native binding loader:** A thin `napi.ts` module loads the native binding
   via the generated `index.js` and types it from the generated `index.d.ts`
   using `typeof import(...)`. Callers (`interpret.ts`, `diagnostic.ts`,
   `index.ts`) import the binding from `napi.ts` and handle `ImageData` dispatch
   directly.

### Bridge Strategy: JSON Round-Tripping

The key TypeScript types exposed from `ballot-interpreter` are JSON-serialized
Rust structs (e.g. `InterpretedBallotCard`, `TimingMarks`, `InterpretError`).
Under Neon, these are serialized to JS via `serde_json` and declared separately
in TypeScript. Under napi-rs, the same JSON round-tripping pattern is preserved:
functions accept and return `serde_json::Value`, with `ts_args_type` and
`ts_return_type` annotations on the `#[napi]` attribute to reference the
existing TypeScript types from `types.ts`. This avoids the need to annotate
every nested Rust struct with `#[napi(object)]` and preserves the existing
TypeScript type surface exactly. See **Prototype Findings** below for the full
comparison.

Function signatures are auto-generated by the `#[napi]` attribute. A
`dts-header.d.ts` file provides imports for the existing TypeScript types
(`BridgeInterpretResult`, `TimingMarks`, `Election`) and defines the
`BridgeInterpretOptions` interface that the generated `index.d.ts` references.

## Alternatives Considered

**Keep Neon.** Neon v1.x introduced the `#[neon::export]` ergonomics that close
some of the gap with napi-rs, and the library reached a stable 1.0 milestone.
However, napi-rs has far greater adoption among high-profile packages (Biome,
Rolldown, Rspack, Oxc, Parcel, SWC, and others) and is more actively releasing
(napi-rs has many recent releases; neon's latest stable is v1.1.1 from May 2024
with v1.2.0 still in alpha as of early 2026). napi-rs also generates the binary
loader and function signatures automatically, eliminating the hand-maintained
`addon.ts`. (Note: napi-rs can also auto-generate TypeScript types for Rust
structs via `#[napi(object)]`, but this migration uses JSON round-tripping
instead, so the hand-written `types.ts` is preserved — see Prototype Findings.)

**Use Wasm/WASI instead of a native addon.** Compiling to WebAssembly avoids the
Node-API layer entirely and would be portable across runtimes. However, image
processing in Wasm incurs a meaningful performance penalty and memory copies
compared to a native addon that can operate directly on Node `Buffer` data. The
ballot interpreter is performance-sensitive (it runs on every scanned ballot),
so this trade-off is not favorable.

**Use FFI via `node-ffi-napi`.** A C-compatible shared library exposed via FFI
avoids any Rust/Node bridge library. This is significantly more complex to set
up and maintain correctly (manual memory management, no automatic type
generation) and is not used anywhere else in the codebase.

## Prototype Findings

Two approaches were prototyped on separate branches to answer whether Rust
structs should use direct `#[napi(object)]` exposure or preserve `serde_json`
serialization:

- `prototype/napi-rs-napi-object` — mirror structs with `#[napi(object)]`
- `prototype/napi-rs-json` — JSON round-tripping with
  `ts_return_type`/`ts_args_type`

### Code size

The napi-object approach more than doubles the bridge file (`js/mod.rs`). Every
type that crosses the boundary requires a `Napi`-prefixed mirror struct
annotated with `#[napi(object)]`, plus a `From` impl converting from the real
type. There are 16 such structs in the prototype.

The JSON approach keeps the bridge file close to the neon baseline. The changes
are mechanical: replace `neon::types::extract::{Error, Json}` with
`napi::bindgen_prelude::Buffer` and `napi_derive::napi`, replace
`#[neon::export]` with `#[napi(...)]`, and add `from_json`/`to_json` helpers for
`serde_json::Value` conversion. The serde structs (`JsInterpretOptions`,
`JsInterpretResult`, `JsInterpretErr`) are kept as-is. Functions that read from
file paths use `tokio::fs::read` for non-blocking I/O, since napi-rs `async fn`
runs on the tokio runtime (which napi-rs bundles via its `async` feature).

### Generated `index.d.ts`

**napi-object** (135 lines): All types are auto-generated as `Napi`-prefixed
interfaces. The names differ from the existing TypeScript types in `types.ts`,
so callers would need to be updated. The `NapiInterpretResult` type loses the
discriminated-union shape from the Rust serde representation — it becomes a flat
struct with an optional `value` field and a string `type` discriminant, which is
weaker than the `Result<InterpretedBallotCard, InterpretError>` alias in
`types.ts`. The napi-object function signatures are synchronous (no `Promise`
wrapping) since they aren't `async fn`.

**JSON**: Only function declarations are generated. Return types reference
`BridgeInterpretResult` and `TimingMarks` imported from `types.ts` via a
`dts-header.d.ts` file. This preserves the existing TypeScript type surface
exactly — callers require no changes to their type imports.

```ts
// napi-object index.d.ts (excerpt)
export declare function interpretPaths(
  electionJson: string,
  sideAImagePath: string,
  sideBImagePath: string,
  optionsJson: string
): NapiInterpretResult;

export interface NapiInterpretResult {
  type: string;
  value?: NapiInterpretedBallotCard;
  isBubbleBallot?: boolean;
  error?: string;
}
// ... 16 more interface declarations
```

```ts
// JSON index.d.ts (excerpt, after dts-header.d.ts prepend)
import type {
  BridgeInterpretResult,
  TimingMarks,
} from './src/bubble-ballot-ts/types';
import type { Election } from '@votingworks/types';

export interface BridgeInterpretOptions {
  /* ... */
}

export declare function interpretPaths(
  election: Election,
  sideAImagePath: string,
  sideBImagePath: string,
  options: BridgeInterpretOptions
): Promise<BridgeInterpretResult>;
// ... other function declarations
```

### Trade-offs

| Concern                  | napi-object                                                | JSON                                       |
| ------------------------ | ---------------------------------------------------------- | ------------------------------------------ |
| Type fidelity            | Medium — `Napi`-prefixed types, flat result structs        | High — preserves existing TS types exactly |
| Annotation burden        | High — 16 mirror structs, 16 `From` impls                  | Low — 4 `ts_return_type` annotations       |
| `types.ts` still needed? | No (but types diverge from existing ones)                  | Yes (but that's already true today)        |
| Caller changes required? | Yes — all callers use `Napi*` types                        | No — same `HmpbInterpretResult` etc.       |
| Serialization cost       | None — direct struct pass-through                          | One JSON round-trip per call               |
| Enum support             | Limited — napi-rs can't represent tagged unions as objects | Full — serde handles arbitrary enums       |
| `async fn` support       | Requires wrapping in `AsyncTask`                           | Native — `async fn` maps to `Promise`      |

### Recommendation

The **JSON approach** is recommended for the migration. The key reasons:

1. **Zero caller changes.** The existing `types.ts` type surface is preserved
   exactly. All 350 lines of `types.ts` remain valid and continue to be the
   source of truth for TypeScript consumers.

2. **Minimal bridge code.** The neon patterns map almost 1:1 to napi-rs
   equivalents, with small additions for `from_json`/`to_json` helpers and async
   I/O for file reads.

3. **Enum limitations in napi-object.** napi-rs does not support Rust enums as
   `#[napi(object)]` types. `InterpretError` is a complex tagged union with many
   variants; representing it in napi-object requires either flattening it to a
   weakly-typed struct (losing the discriminated union) or leaving it as JSON
   anyway — negating the benefit of the approach for the most important type.

4. **Serialization cost is negligible.** The ballot interpreter is CPU-bound on
   image processing. The JSON serialization of a result struct adds microseconds
   to a call that takes hundreds of milliseconds.

With JSON round-tripping, `types.ts` remains the authoritative TypeScript type
source and is still needed by callers. The `addon.ts` file — which previously
handled binary loading, function typing, and `ImageData` dispatch — is fully
replaced: binary loading moves to the napi-rs generated `index.js`, function
types are derived from the generated `index.d.ts` via `typeof import(...)`, and
`ImageData` dispatch is inlined into the callers (`interpret.ts`, `index.ts`,
`diagnostic.ts`).
